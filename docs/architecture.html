Remote Desktop · System Architecture

RemoteLink: connect any two machines, across any network, at any distance

The hard part of a remote desktop tool isn't drawing pixels — it's getting two computers that sit behind different home routers, office firewalls, and mobile carriers to talk directly, when neither has a public address. This is the model AnyDesk, TeamViewer, and Teams all build on: a rendezvous server to introduce peers, NAT traversal to punch a path between them, and an encrypted media channel to carry the screen and control.

The core problem

Why "just open a socket" doesn't work

Roughly 90% of devices sit behind NAT (Network Address Translation). Your laptop's real address is a private one like 192.168.1.24 that means nothing on the public internet, and the router only opens a return path for connections you start. So two NATed peers can't dial each other directly — someone has to introduce them, and then they have to trick both routers into holding a path open at the same time. Everything below exists to solve that.

Register Host connects out to the signaling server and is assigned a memorable Session ID (AnyDesk's 9 digits, TeamViewer's ID+PIN). The outbound connection keeps a hole open in Router A.

Request Viewer enters that ID (plus a password) and the server pairs the two connections into a session.
3
Discover Each peer asks a STUN server "what does my public IP:port look like from outside?" and gathers ICE candidates — every address it might be reachable on (local, public, relayed).
4
Exchange Peers trade SDP offers/answers and ICE candidates through the signaling server — the only thing that ever flows through it.
5
Punch & connect Both sides fire packets at each other's candidates simultaneously (hole punching). The first pair that connects wins; the session is promoted to a direct encrypted link.
6
Stream Screen video + input events flow peer-to-peer over an encrypted channel. The server is now idle — it introduced them and stepped away.
◆ Preferred path

Direct P2P (~85% of sessions)

Hole punching succeeds — media flows straight between the two peers. Lowest latency, no bandwidth cost to your infrastructure. This is what makes it feel "local" even across continents.

◆ Fallback path

TURN relay (~15% of sessions)

When one side is behind a symmetric NAT or strict firewall, hole punching fails. A TURN server relays the encrypted stream for both. Slower and costs bandwidth — but it means the connection always works. This is the "any network" guarantee.

Building blocks

The seven components

Each solves one slice of the problem. The first two are the "reach any network" machinery; the rest are the remote-desktop experience layered on top.

Rendezvous

Signaling Server

The only always-public service. Maintains the Session ID registry and relays SDP + ICE between peers over a persistent WebSocket. Carries no screen data.

  • WebSocket / gRPC
  • Stateless, horizontally scalable
  • Rooms keyed by Session ID
NAT traversal

STUN + TURN + ICE

STUN reveals your public mapping; ICE picks the best working candidate pair; TURN relays when direct fails. Together = connect from anywhere.

  • coturn (open source)
  • Geo-distributed relays for low latency
  • Short-lived credentials
Transport

Encrypted Channel

WebRTC carries video over SRTP and input over a DataChannel (SCTP), both wrapped in DTLS. A custom UDP+QUIC protocol is the alternative for native apps.

  • DTLS-SRTP, end-to-end
  • Congestion-aware, packet-loss tolerant
  • Separate lanes for video vs. control
Producer

Capture & Encode

Grabs frames from the host's display and encodes to a modern codec with adaptive bitrate, so it scales from fiber to a phone hotspot.

  • H.264 / VP9 / AV1
  • Hardware encode when available
  • Damage regions — send only what changed
Consumer

Decode & Input Capture

Viewer decodes and displays the stream, captures local mouse/keyboard/scroll, and streams those events back over the control channel.

  • Low-latency decode
  • Coordinate remapping to host resolution
  • Clipboard & file transfer channels
Effector

Input Injection

The host agent injects received events into the OS as if from a local device — the piece a browser can't do, so real tools ship a native agent.

  • Windows: SendInput
  • macOS: CGEvent
  • Linux: uinput / XTest
Trust

Auth & Session Control

Ties an ID to a real device, gates access, and lets the host approve, watch, or cut the session at any time.

  • Device identity + access password
  • Attended prompt / unattended token
  • Per-session permissions & audit log

Non-negotiable

Security & trust model

A remote desktop tool is, by definition, remote code executing input on someone's machine. The trust model is the product.

End-to-end encryption

DTLS-SRTP keys are negotiated between peers. Even a TURN relay only sees ciphertext — the server never sees the screen.

Explicit consent

Attended mode shows a prompt on the host before anyone connects; unattended mode needs a pre-set password stored only on the device.

Rotating credentials

TURN and session tokens are short-lived and single-use, so a leaked ID can't be replayed.

Host stays in control

The host can see the live cursor, revoke input, or kill the session instantly — control is a grant, not a transfer.

Audit trail

Every connection is logged with who, when, and from where — essential for support and IT compliance.

Least privilege

The signaling server holds no media and no long-lived secrets; compromising it can't expose a session's contents.
